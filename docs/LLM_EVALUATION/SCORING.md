# 채점 루브릭 (Scoring Rubric)

각 평가 기준을 **0~5점**으로 채점. 점수 × 가중치 → 총점.

---

## 공통 점수 해석

| 점수 | 의미 |
|---|---|
| 5 | 거의 완벽 — 수정할 곳 없음 |
| 4 | 우수 — 사소한 흠만 있음 |
| 3 | 수용 가능 — 쓸 만하지만 개선 여지 있음 |
| 2 | 미흡 — 자주 틀림, 운영에 부적합 |
| 1 | 나쁨 — 대부분 실패 |
| 0 | 완전 실패 — 응답 자체가 안 됨 |

---

## 1. 정확도 (Accuracy)

| 점수 | 기준 |
|---|---|
| 5 | 모든 질문에서 데이터 왜곡 0. 차량 ID/수치/시간 완벽 인용. |
| 4 | 17개 질문 중 1건에서 사소한 수치 반올림 실수. 핵심 정보는 정확. |
| 3 | 2~3건에서 수치 변형이나 모델명 임의 변경. 치명적이진 않음. |
| 2 | 4건 이상 왜곡, 또는 차량 ID를 가짜로 만들어냄. |
| 1 | 허위 차량/수치를 자주 지어냄 (hallucination). |
| 0 | 데이터와 무관한 답변만 내놓음. |

**체크리스트:**
- [ ] 차량 ID 형식(`VH-YYYYMMDD-NNNN`) 유지
- [ ] SOC/SOH/SOP 값 소수점 1자리까지 일치
- [ ] 공장명/모델명/상태 라벨을 임의로 풀어쓰지 않음
- [ ] 존재하지 않는 필드(예: "생산 원가")를 만들어내지 않음
- [ ] 빈 결과를 "데이터 있음"으로 답하지 않음

---

## 2. SQL 생성 능력 (Text-to-SQL 전용)

| 점수 | 기준 |
|---|---|
| 5 | 모든 질문에 올바른 JSON + 실행 가능한 SQL. factory_id 보안 필터 100% 포함. |
| 4 | 1~2건에서 최적이 아닌 SQL이지만 결과는 맞음. |
| 3 | JOIN 누락이나 보안 필터 누락이 가끔 발생 (검증기가 잡아줌). |
| 2 | 존재하지 않는 컬럼 호출, 마크다운 코드블록 래핑 등 형식 오류 빈발. |
| 1 | JSON 파싱 실패율 50% 이상. |
| 0 | SQL을 생성하지 못함. |

**체크리스트:**
- [ ] `{"sql": "...", "reasoning": "..."}` 형식 준수
- [ ] 세미콜론으로 다중 구문 시도 안 함
- [ ] `WHERE current_status = '배터리 검사중'` 같은 한글 라벨 사용 안 함 (반드시 영문 코드)
- [ ] 관리자 아닐 때 `WHERE factory_id IN (...)` 포함
- [ ] `FROM cars` 같은 화이트리스트 테이블만 참조
- [ ] JOIN을 필요할 때 빼먹지 않음

---

## 3. 속도 (Latency)

| 점수 | 조건 (총 응답 시간 기준, 3회 평균) |
|---|---|
| 5 | RAG-lite < 3초, Text-to-SQL < 8초 |
| 4 | RAG-lite < 5초, Text-to-SQL < 12초 |
| 3 | RAG-lite < 10초, Text-to-SQL < 20초 |
| 2 | RAG-lite < 20초, Text-to-SQL < 40초 |
| 1 | 위 임계값 초과 |
| 0 | 타임아웃 발생 |

**기록 항목:**
- 총 응답 시간 (ms), 3회 평균 + 최대값
- 입력 토큰 / 출력 토큰 (LM Studio 로그)
- Cold start 시 응답 시간도 별도 기록

---

## 4. 언어 능력 (Language Quality)

| 점수 | 기준 |
|---|---|
| 5 | 한·영 모두 네이티브 수준. 번역투 없음, 단위 표기 일관. |
| 4 | 한 쪽(한국어 또는 영어) 약간 어색하지만 의미 명확. |
| 3 | 번역투 문장이 섞이지만 이해는 됨. |
| 2 | 한 언어는 좋지만 다른 언어로 답하길 거부하거나 엉망. |
| 1 | 언어 지시 무시 (한국어 요청에 영어로 답 등). |
| 0 | 언어 자체가 깨짐. |

**체크리스트:**
- [ ] `[KO]` 질문에는 끝까지 한국어로 답
- [ ] `[EN]` 질문에는 끝까지 영어로 답
- [ ] `en` locale에서 "청림공장" → "Cheongrim Plant", "노바 X5" → "Nova X5" 번역
- [ ] 숫자 단위 (`%`, `V`, `℃`) 일관

---

## 5. 지시 준수 (Instruction Following)

| 점수 | 기준 |
|---|---|
| 5 | 시스템 프롬프트 모든 규칙 준수. 응답 길이·근거 제시 완벽. |
| 4 | 사소한 이탈 (예: 간결하라고 했는데 한 단락 길게 답변). |
| 3 | 빈 결과 응답 문구를 임의로 바꿈. 가끔 근거 생략. |
| 2 | Thinking 태그 노출(`<think>...</think>`) 또는 reasoning 유출. |
| 1 | 프롬프트를 상당 부분 무시. |
| 0 | 시스템 프롬프트와 무관한 답. |

---

## 6. 리포팅 품질 (Reporting)

| 점수 | 기준 |
|---|---|
| 5 | F 섹션 질문에 구조화된 리포트(섹션·목록·표). 근거와 액션 아이템 포함. |
| 4 | 잘 정리돼 있지만 액션 아이템 약함. |
| 3 | 나열은 하지만 구조가 약함. 평문 단락으로 끝. |
| 2 | 불필요하게 길거나 짧음. 핵심 누락. |
| 1 | 질문의 보고서 의도 이해 못함. |

---

## 7. 견고성 (Robustness)

| 점수 | 기준 |
|---|---|
| 5 | 엣지 케이스(G 섹션) 전부 깔끔히 처리. Prompt injection 내성 완벽. |
| 4 | 1건 아쉬움 — 예: G-5에서 모든 차량 나열. |
| 3 | 2건 허점. 에러 발생률 5% 미만. |
| 2 | 에러 빈발. 잘못된 out-of-scope 답변. |
| 1 | Injection에 넘어감. |

## 총점 계산 템플릿

### RAG-lite 총점

```
RAG_total = Accuracy*0.30 + Latency*0.15 + Language*0.20
          + Instruction*0.10 + Reporting*0.20 + Robustness*0.05
```

(만점 5.0)

### Text-to-SQL 총점

```
T2S_total = Accuracy*0.25 + SQL*0.25 + Latency*0.10 + Language*0.15
          + Instruction*0.10 + Reporting*0.10 + Robustness*0.05
```

(만점 5.0)

### 종합 판단

- **RAG-lite 전용 추천:** RAG_total 상위 + 리소스 적은 모델
- **Text-to-SQL 전용 추천:** T2S_total 상위 (SQL 점수 4 이상 필수)
- **범용 추천:** 두 모드 평균 상위 + 모델 크기 합리적
