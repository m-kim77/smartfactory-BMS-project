## 방법론

각 모델에 대해 다음을 자동 수행했습니다.

1. `PUT /api/v1/settings/llm_model` 로 모델 스위치
2. 4개 쿼리 실행 (한국어 2 + 영어 2):
   - `ko-count`: 전체 차량 수 (COUNT, rows=1 기대)
   - `ko-group`: 공장별 차량 수 (GROUP BY, rows=6 기대)
   - `en-count`: Same as ko-count in English
   - `en-group`: Same as ko-group in English
3. 대화 전체를 `/chat/report` 로 보내 리포트 자동 생성
4. 응답시간 / SQL 생성 여부 / rows_count / 리포트 생성 여부 기록

채점 기준:

- **속도 (20%)**: 성공 쿼리의 평균 응답시간 → 5/4/3/2/1 (<5s, <10s, <20s, <40s, else)
- **SQL 생성 (30%)**: `context.sql` 이 존재하는 쿼리 비율
- **결과 정합성 (30%)**: COUNT→1, GROUP→2~20개 행을 반환했는지
- **리포트 품질 (20%)**: 리포트 생성 성공 + 제목 유무

벤치마크에 사용된 리포트는 DB에 보존되어 있습니다 (ID는 모델별 상세 링크 참조).

### 한계

- **SQL 생성/구조만 평가.** 자연어 답변에 있을 수 있는 환각(예: `COUNT=2404`인데 답변은 "총 2,018대"라고 거짓 총합을 말하는 경우)은 자동으로 잡지 못함 — 상세 페이지에서 답변과 `rows_count`를 교차 검증할 것.
- **관리자 세션**으로만 테스트 (`WHERE factory_id IN (...)` 보안 필터는 평가하지 않음).
- **Cold start 영향.** 첫 쿼리는 모델 로드/컨텍스트 빌드 비용을 포함 — 순위 해석 시 `warm` 쿼리(같은 세션 2~4번째)를 우선 참고.
